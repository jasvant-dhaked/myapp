1 Executive Summary (Non-Technical)

Imagine you tap a link someone sends you on WhatsApp. It looks like a normal link. Your phone opens the FamPay app automatically, which seems normal. But silently, in the background, the app hands over your account login token, your live GPS coordinates, and your entire phone contacts list to the attacker — all without showing any warning or asking for any permission.


This is possible because FamPay opens a special internal browser ("WebView") to load web pages, and this browser is given the ability to access private phone data. The problem is: the app does not check whose webpage it is loading before handing over these powers. An attacker can trick the app into loading their own malicious webpage and immediately steal the user's data.


Who is at risk? Every FamPay user on Android. No special knowledge or action is required from the victim beyond tapping a link.

3 Technical Root Cause

Root Cause A — DeepLinkActivity routes arbitrary URLs into WebviewActivity

DeepLinkActivity is exported (reachable by any app or link on the device). It accepts the deeplink scheme fp://fampay.in/home_init/ with a base64-encoded URL in the init_deeplink parameter. This URL is decoded and stored globally — then passed to WebviewActivity with no domain validation.

Root Cause B — JavaScript interfaces injected into every URL, including attacker pages

WebViewFragment calls addJavascriptInterface() for all 4 interfaces unconditionally — it does not check whether the loaded URL is a trusted FamPay domain. The NativeAuthAndroid interface (which exposes the auth token) is added when fp_auth_req=true or fp_type=fpSurvey is in the URL — both of which the attacker controls.

5 Full Attack Chain
1
Attacker crafts malicious link

Creates a URL: fp://fampay.in/home_init/?init_deeplink=<base64(attacker_page?fp_type=fpSurvey&fp_auth_req=true)>

2
Victim taps the link

Sent via SMS, WhatsApp, email, or embedded in a webpage. Android prompts to open with FamPay.

3
DeepLinkActivity decodes URL

Base64 is decoded → attacker URL stored globally → HomeActivity launched.

4
WebviewActivity loads attacker page

Attacker's page (e.g. ngrok URL) opens inside FamPay's internal authenticated browser.

5
JavaScript interfaces injected

Because URL contains fp_type=fpSurvey, all 4 interfaces including NativeAuthAndroid are injected into window.

6
Attacker JavaScript calls native APIs

NativeAuthAndroid.getAuthToken() → auth token
NativeAuthAndroid.getUserDetails() → PII
NativeActionsAndroid.getCurrentLocation() → GPS
ContactActionsAndroid.getNativeContacts() → contacts

6 Remediation (For Developers)
1
Allowlist trusted domains before injecting interfaces. Before calling addJavascriptInterface(), check that the URL being loaded has a host that ends in .fampay.in or .famapp.in. Reject all other origins.
2
Never trust URL parameters for security decisions. The y1() method grants NativeAuthAndroid based on fp_auth_req=true in the URL — an attacker-controlled value. This check must be based on the domain of the loaded page, not its parameters.
3
Validate the decoded deeplink URL in DeepLinkActivity. After base64-decoding init_deeplink, verify the host is in an explicit allowlist before storing or routing it. Reject null, non-HTTPS, and non-FamPay domains.
4
Remove NativeAuthAndroid from WebView entirely. Auth tokens should never be accessible from a WebView. Use session cookies with httpOnly + secure flags, or pass tokens via server-side rendering only.
Recommended fix — WebViewFragment.java (pseudo-code)
// Before adding interfaces, validate the URL host String host = Uri.parse(loadedUrl).getHost(); boolean isTrusted = host != null && (host.endsWith(".fampay.in") || host.endsWith(".famapp.in")); if (isTrusted) { this.webview.addJavascriptInterface(new WebViewActionsJavascriptInterface(...), "NativeActionsAndroid"); this.webview.addJavascriptInterface(new JSContactsBridge(), "ContactActionsAndroid"); // Only add auth interface for trusted, authenticated origins if (y1()) { this.webview.addJavascriptInterface(new WebViewAuthJavascriptInterface(...), "NativeAuthAndroid"); } }
