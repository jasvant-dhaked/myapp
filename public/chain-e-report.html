Executive Summary (Non-Technical)

FamPay uses Razorpay for payment processing. There is a flaw in how FamPay loads the Razorpay checkout screen: it uses a method called loadDataWithBaseURL that sets the page's "origin" to api.razorpay.com — even when the actual HTML content is controlled by an attacker.

This is like printing a fake bank statement on official letterhead. The paper (origin/URL) looks real because it says "api.razorpay.com", but the content — including the "Enter your UPI PIN" form — is entirely controlled by the attacker. The user has no way to tell the difference.

When chained with the deeplink delivery vulnerability (Chain A), an attacker can trick any FamPay user into seeing a convincing payment confirmation screen that steals their UPI PIN, card number, or OTP.

Impact

UPI PIN Theft
Fake payment screen captures 4 or 6-digit UPI PIN — enables unauthorized transfers
Card Credential Theft
Fake checkout form captures card number, CVV, expiry
OTP Interception
Attacker controls the "Enter OTP" form, captures 2FA codes in real time
Zero Suspicion
Rendered inside the official FamPay app with Razorpay branding — passes visual inspection

Vulnerable Code

BaseRazorpay.java (Lines 1532–1536)

// strReplace contains attacker-influenced content
webView.loadDataWithBaseURL(
    "https://api.razorpay.com",   // ← sets origin to trusted Razorpay domain
    strReplace,                    // ← HTML content (attacker-controlled)
    "text/html",
    "UTF-8",
    null
);

The strReplace variable includes data from the original Intent/deeplink parameters. If an attacker can influence those parameters (via Vuln #2 deeplink delivery), they control the HTML rendered with https://api.razorpay.com as the trusted base URL.

Remediation

Fix 1 — Never use loadDataWithBaseURL with user/deeplink-influenced content

// BAD: user-controlled HTML with trusted origin
webView.loadDataWithBaseURL("https://api.razorpay.com", userContent, ...);

// GOOD: load from the actual Razorpay URL
webView.loadUrl("https://api.razorpay.com/checkout/embedded");

// Or: use loadData() without a base URL (origin = null/about:blank)
webView.loadData(encodedHtml, "text/html", "UTF-8");

Fix 2 — Sanitize all deeplink parameters before rendering

// Allowlist approach: only permit known-safe checkout parameters
String checkoutData = intent.getStringExtra("checkout_data");
if (checkoutData != null && isValidCheckoutJson(checkoutData)) {
    // Only pass to Razorpay SDK, never inject into raw HTML
    razorpay.open(parseCheckoutOptions(checkoutData));
}

Fix 3 — Update / replace Razorpay SDK

Contact Razorpay to confirm if a patched SDK version is available that does not use loadDataWithBaseURL with injectable content. Pin the SDK version and monitor for security patches.
